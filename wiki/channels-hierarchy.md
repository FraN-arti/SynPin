# 🏢 Каналы и иерархия

SynPin — не просто набор агентов. Это **организация** с каналами, ролями, главами отделов и советом директоров.

---

## Почему это уникально

Существующие фреймворки:
- **CrewAI** — агенты работают последовательно, нет общения
- **AutoGen** — агенты общаются, но без структуры и модерации
- ** delegate_task** — одноразовые агенты без памяти

**SynPin** — агенты работают как компания:
- Отделы с каналами
- Главы отделов модерируют
- Совет отделов решает кросс-функциональные задачи
- Совет директоров — стратегия
- Память каждого агента + общий контекст

---

## Уровни организации

```
Уровень 3:  Совет директоров (Board of Directors)
             └── Стратегия, приоритеты, бюджет
                     │
Уровень 2:  Совет отделов (Heads Council)
             └── Кросс-функциональные задачи, делегирование
                     │
Уровень 1:  Каналы отделов (Department Channels)
             └── Рабочие агенты, выполнение задач
```

## Каналы отделов

Каждый отдел — отдельный канал с контекстом, участниками и лентой.

### Структура канала

```
~/.synpin/data/channels/
├── web-dev/
│   ├── context.md           # что делает этот отдел
│   ├── chat-log.md          # лента сообщений
│   ├── decisions.md         # принятые решения
│   └── members.yaml         # участники + роли
├── api-design/
│   ├── context.md
│   ├── chat-log.md
│   ├── decisions.md
│   └── members.yaml
└── qa-testing/
    ├── context.md
    ├── chat-log.md
    ├── decisions.md
    └── members.yaml
```

### members.yaml

```yaml
channel: "web-dev"
department: "Web Development"

head: "architect"        # глава отдела

members:
  - name: "frontend"
    role: "worker"
    model: "default"
    tools: ["code", "files"]

  - name: "developer"
    role: "worker"
    model: "default"
    tools: ["code", "search", "files"]

  - name: "designer"
    role: "worker"
    model: "default"
    tools: ["code", "search"]
```

### context.md

```markdown
# Channel: web-dev

## Purpose
Разработка и поддержка веб-интерфейса SynPin.

## Tech Stack
- React 19 + Vite 6 + TypeScript
- Tailwind 4
- Zustand для state management

## Active Projects
- Форма авторизации (в процессе)
- Страница агентов (в очереди)

## Standards
- Чистый CSS, не Tailwind (предпочтение пользователя)
- Zod для валидации форм
- Тесты обязательны для API-интеграций
```

## Роли

### Глава отдела (Head)

Управляющий каналом. Его задачи:

| Действие | Пример |
|---|---|
| **Модерировать** | `⏸️ Хватит спорить, решение принято` |
| **Делегировать** | `@developer, сделай X. Дедлайн: сегодня` |
| **Перенаправить** | `Это не для нас. Передаю в api-design` |
| **Резюмировать** | `Итого: решили X, потому что Y` |
| **Утвердить** | `✅ Утверждено. Работаем` |
| **Отклонить** | `❌ Не подходит. Причина: Z` |

### Работник (Worker)

Исполнитель. Его задачи:
- Выполнять задачи от главы
- Предлагать решения
- Обращаться к коллегам через `@mention`
- Записывать ошибки в свою память

### Директор (Director)

Стратег. Его задачи:
- Определять приоритеты
- Распределять ресурсы между отделами
- Принимать финальные решения при конфликтах
- Обновлять shared MEMORY.md

## Совет отделов

Канал для глав отделов. Здесь решаются кросс-функциональные задачи.

### Пример: MCP сервис авторизации

```
[Канал: heads-council]

@head-api: Коллеги, на сервер добавлен MCP сервис с авторизацией.
           Точка входа: POST /api/v1/auth/login
           Документация обновлена в shared/memory.md

@head-web: Принято. Передаю команде — сделаем форму логина.

@head-qa: Добавлю тесты на авторизацию. Нужен ли rate limiting?

@head-api: Да, 10 запросов/мин. Уже настроено на сервере.

@head-web: ✅ Понял. Забираю задачу.
```

### Результат

```
[Канал: web-dev]

@head-web: Команда, новая задача — форма авторизации.
           MCP endpoint: POST /api/v1/auth/login
           Rate limit: 10 req/min
           Документация в shared/memory.md

@frontend: Сделаю форму с email + password + валидация через zod

@developer: Возьму интеграцию с API и refresh token flow

@head-web: ✅ Распределил. Работаем.
```

## Совет директоров

Высший уровень. Стратегические решения.

### members.yaml

```yaml
channel: "board"
type: "board-of-directors"

members:
  - name: "architect"
    role: "director"
    focus: "Техническая стратегия"

  - name: "strategist"
    role: "director"
    focus: "Бизнес-приоритеты"

  - name: "product"
    role: "director"
    focus: "Пользовательские потребности"
```

### Типичные задачи

| Задача | Кто решает |
|---|---|
| Какой фичей заняться следующей? | Совет директоров → Совет отделов → Отделы |
| Конфликт между отделами | Совет директоров (финальное решение) |
| Изменение стека технологий | Совет директоров → все отделы |
| Приоритизация бэклога | Совет директоров |

## Cross-channel делегирование

Когда задача затрагивает несколько отделов:

```
1. Совет директоров: «Нужна авторизация на всех уровнях»
       ↓
2. Совет отделов: head-api берёт сервер, head-web берёт клиент
       ↓
3. Канал api-design: developer реализует endpoint
       ↓
4. Канал web-dev: frontend делает форму, developer интегрирует
       ↓
5. Канал qa-testing: reviewer тестирует весь flow
       ↓
6. Совет отделов: «Готово, закрываем задачу»
```

## @mention — обязательное обращение

В канале агенты **обязаны** обращаться друг к другу через `@`:

```
✅ @developer, можешь проверить мой код?
✅ @head-web, задача готова к ревью
❌ "проверь код" (неясно кому)
❌ "задача готова" (неясно кому сообщают)
```

Это даёт:
- Чёткую адресацию — каждый знает когда его вызывают
- Контекст для главы — видно кто с кем общается
- Фильтрацию — агент реагирует только на свои `@`

## Конфигурация каналов

Все каналы настраиваются через `~/.synpin/config/channels.yaml`:

```yaml
channels:
  web-dev:
    department: "Web Development"
    head: "architect"
    members: ["frontend", "developer", "designer"]
    tools: ["code", "files", "search"]

  api-design:
    department: "API Design"
    head: "architect"
    members: ["developer", "researcher"]
    tools: ["code", "search", "files"]

  heads-council:
    type: "council"
    members: ["head-api", "head-web", "head-qa"]

  board:
    type: "board"
    members: ["architect", "strategist", "product"]
```

## КПД = память + контекст + иерархия

Каждый агент в канале:

1. **Видит** все сообщения канала (channel context)
2. **Знает** свои прошлые ошибки (personal MEMORY.md)
3. **Знает** ошибки коллег (shared MEMORY.md)
4. **Слышит** главу (moderation)
5. **Обращается** через `@` (чёткая адресация)

Результат: **агенты не повторяют ошибки, не спорят бесконечно, работают как команда**.
